# AGENTS.md

## Build And Test
- Despite the repo name, this checkout has both CMake and Bazel builds. CI uses CMake: `cmake --preset <preset>`, `cmake --build --preset <preset>`, and `ctest --preset <preset>`. The Bazel build is a parallel addition and does not modify any C++ sources or CMake files.
- First clone step is `git submodule update --init --recursive`; default builds expect the `ThirdParty/{asio,fmt,googletest,oatpp,spdlog}` submodules to exist.
- Presets write to `_build/<preset>` and `_install/<preset>`. Common local presets: `debug`, `release`, `relwithdebinfo`. `distrib` is the packaging preset and turns on warnings-as-errors, source install, and symbols packaging.
- Focused verification: `ctest --preset debug --output-on-failure -R '^Test_Uri$'`. CTest names come from source filenames: `Test_*`, `ITest_*`, `FTest_*`.
- CI only runs `ctest --preset <preset> --output-on-failure -R '^(I|T)' --timeout 300`, so `FTest_*` functional/perf suites do not run unless you invoke them explicitly.
- Most CTest integration/functional suites use `SilKit/IntegrationTests/SimTestHarness`, which starts an in-process registry and system controller. Do not start `sil-kit-registry` manually for those tests.
- For manual demo/app runs instead, start `sil-kit-registry`, then `sil-kit-system-controller <participants...>`, then the executable.

## Bazel Build
- Bazel version is **8.2.1** (pinned in `.bazelversion`). Configuration lives in `.bazelrc` and `MODULE.bazel`.
- The Bazel build is parallel to CMake -- it does **not** modify existing C++ sources or CMake files. Keep `BUILD.bazel` files near the corresponding `CMakeLists.txt` where practical.
- Full build: `bazel build //SilKit/source/... //Utilities/...`
- Full test: `bazel test //SilKit/... //Utilities/... //bazel:smoke_test --test_output=errors --test_timeout=120`
- Release build (optimized + stripped executables): `bazel build --config=release //SilKit/source:SilKit //Utilities/...`
- Dashboard toggle: `--config=no_dashboard` disables dashboard (default is ON).
- **146 non-manual tests pass** (145 unit/integration + 1 smoke test). 7 FTests are tagged `manual`.
- After any major change, always run the full test suite to verify no regressions.

### Bazel Architecture
- Cyclic CMake interface libraries (~20 `I_SilKit_*` targets) are merged into a single `silkit_internal_headers` target in `SilKit/source/BUILD.bazel`. Non-cyclic leaf targets (util, config, wire) remain in their own packages.
- All `O_*`-equivalent implementation libraries use `alwayslink = True` to match CMake OBJECT library semantics.
- All `cc_test` targets use `linkstatic = True` to avoid dynamic linking issues with `alwayslink` deps.
- `S_SilKitImpl` is the static aggregate of all implementation libraries (used by `sil-kit-registry` and tests).
- `SilKit` is the `cc_shared_library` (uses version script `silkit.map`, `-Wl,-soname,libSilKit.so`, `--exclude-libs=libspdlog.a`).
- `SilKit_dynamic` is a `cc_import` wrapper around the shared library, used by `sil-kit-system-controller` and `sil-kit-monitor` for dynamic linking (matching CMake).
- Version macros are generated by a custom Starlark rule (`bazel/version_macros.bzl`) using workspace status stamping.

### Bazel Utility Linking
- `sil-kit-registry`: statically links `S_SilKitImpl` + dashboard (matching CMake). Defines `EXPORT_SilKitAPI`. Uses C++17.
- `sil-kit-system-controller`: dynamically links `libSilKit.so` via `SilKit_dynamic` (matching CMake). Does NOT define `EXPORT_SilKitAPI`.
- `sil-kit-monitor`: dynamically links `libSilKit.so` via `SilKit_dynamic` (matching CMake). Does NOT define `EXPORT_SilKitAPI`.
- All three utilities have `-Wl,--strip-all` on Linux (matching CMake Release stripping). The shared library is NOT stripped.

### Bazel Test Infrastructure
- Test main: `SilKit/cmake/SilKitTest/BUILD.bazel` provides `silkit_test_main` (with version print) and `silkit_test_main_no_version` (for hourglass tests that don't link `S_SilKitImpl`).
- Data-file tests: `silkit_data_test` macro in `bazel/cc_defs.bzl` + `bazel/run_with_data.sh` wrapper. Builds as `cc_binary`, wraps with `sh_test`, symlinks data files into a temp CWD.
- SimTestHarness: two variants in `SilKit/IntegrationTests/SimTestHarness/BUILD.bazel` -- public API and internal API (with `SILKIT_LEGACY_CPP_API=1`).
- Integration tests: each ITest/FTest is a separate `cc_test` (unlike CMake mega-executables).

### QNX Cross-Compilation
- Self-contained toolchain: `bazel/qnx/extensions.bzl` generates real repos when `QNX_HOST` is set, stubs when unset.
- Build commands: `bazel build --config=qnx_x86_64 ...` or `--config=qnx_aarch64`.
- Source the QNX SDP first: `source /home/vector/qnx800/qnxsdp-env.sh` (alias: `qnx-env` in interactive shells).
- QNX platforms use `@platforms//os:linux` + `@qnx_cc_toolchain//:qnx`; `//bazel:is_qnx` is a strict superset of `//bazel:is_linux` for correct `select()` specialization.
- QNX-specific defines: `_QNX_SOURCE`, `SA_RESTART=0`, `FD_SETSIZE=1000`. Links `-lsocket` (no `-lpthread`, no `-ldl`).
- `ThirdParty/BUILD.bazel` has a `:qnx` config_setting that matches our platform model (not `@platforms//os:qnx`) to enable `-lregex` linking on googletest. All third-party Bazel targets are centralized there; submodule directories are never modified. Submodule-shipped BUILD files are suppressed via `--deleted_packages` in `.bazelrc`.

### Binary Parity Verification
- Smoke test: `bazel test //bazel:smoke_test` verifies SONAME, C API exports (147 symbols), dynamic deps, and registry start/stop.
- Normalized comparison script: `bazel/binary_parity.sh <cmake_bin> <bazel_bin> <label>` compares ELF headers, dynamic/static symbols, section sizes, NEEDED, SONAME, file sizes.
- Known differences: Bazel produces smaller binaries due to more aggressive `--gc-sections` dead-code elimination (especially visible in the statically-linked registry: ~3.3M Bazel vs ~6.6M CMake).

## Repo Map
- `SilKit/` is the main product code and tests. The library wiring lives in `SilKit/source/` (`core`, `services`, `extensions`, `capi`, `dashboard`).
- `Utilities/` builds the shipped tools: `sil-kit-registry`, `sil-kit-system-controller`, and `sil-kit-monitor`.
- `Demos/` are example apps. When built outside the root project they switch to `find_package(SilKit)` against an installed package.
- `ThirdParty/` is vendored/submodule code; avoid editing it unless the task is explicitly about a bundled dependency. `ThirdParty/BUILD.bazel` is our file (not from any submodule) and centralizes all third-party Bazel targets with QNX platform support.
- `bazel/` contains Bazel helper infrastructure: `cc_defs.bzl` (macros), `version_macros.bzl` (version generation), `run_with_data.sh` (test data wrapper), `smoke_test.sh`, `binary_parity.sh`, `qnx/` (QNX toolchain).
- `bazel-migration-plan.md` tracks milestone progress for the Bazel migration.

## Toolchain Gotchas
- Language standard is not uniform: `SilKit/` and `sil-kit-registry` use C++17, but `Demos/`, `sil-kit-system-controller`, and `sil-kit-monitor` are C++14. The Bazel build sets C++17 globally (superset, works for all).
- `SILKIT_BUILD_DASHBOARD` defaults to `ON` and pulls in bundled `oatpp`. Sanitizer CI and mingw/system-library builds turn it `OFF`; mirror that if your change is unrelated to dashboard and you want CI-like sanitizer coverage. Bazel equivalent: `--config=no_dashboard`.
- `EXPORT_SilKitAPI` controls C API visibility (`__attribute__((visibility("default")))` on GCC). It must be defined when compiling code that *exports* SilKit C API symbols (shared library, registry). It must NOT be defined for code that *imports* them (dynamically-linked utilities, demos).
- Docs are opt-in: `pip3 install -r SilKit/ci/docker/docs_requirements.txt`, configure with `-D SILKIT_BUILD_DOCS=ON`, then build target `Doxygen` (it runs both Doxygen and Sphinx).
- Windows CI pins CMake `3.31.6` because newer hosted-runner CMake versions break the bundled `yaml-cpp` build.
- Bazel `--strip=always` only strips debug symbols (`-Wl,-S`), not the full symbol table. To match CMake Release stripping, use `-Wl,--strip-all` per executable.
- Bazel's `set -euo pipefail` in shell scripts combined with `grep -q` in a pipe causes SIGPIPE failures. Always capture command output into a variable first, then grep the variable.

## Formatting And Licenses
- Formatting check is `python3 ./SilKit/ci/check_formatting.py`; it expects `clang-format-18` and only scans `SilKit/`, `Demos/`, and `Utilities/`.
- `check_formatting.py` still exits `0` when it prints formatting warnings. Do not rely on the exit code alone; inspect warnings or run `clang-format-18` on touched files yourself.
- New `.cpp/.hpp/.h/.c/.py/.sh` files and `CMakeLists.txt` need SPDX headers; `SilKit/ci/check_licenses.sh` enforces this outside `ThirdParty/`.
- Bazel files (`BUILD.bazel`, `.bzl`, `.bazelrc`) also use SPDX headers for consistency.
- `CONTRIBUTING.md` currently says the upstream project is not accepting external pull requests.
